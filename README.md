## Задание 5
Разработать схему ML-системы для цифровой рекламы, встраивающей бренды в видеопоток в реальном времени.

Ответы на вопросы задания
1. Какова природа и объём данных?

По идее система работает с несколькими типами данных:
- неструктурированные данные — видеопотоки и кадры;
- структурированные данные — параметры рекламных кампаний, настройки таргетинга, данные о зрителе;
- полуструктурированные данные — логи событий, метрики показа и работы системы.  
Однако основной объём создаёт видео, так как оно занимает много памяти и требует вычислений, но при этом видеоданные хорошо 
распараллеливаются: их можно обрабатывать по сценам, сегментам и кадрам.

2. Какова требуемая скорость обработки?  
Для показа рекламы нужна обработка почти в реальном времени. Система должна успеть:
- выбрать подходящий бренд;
- определить место вставки;
- встроить бренд в видео;
- отправить персонализированный сегмент зрителю.  
Поэтому для online-контура используется потоковая обработка (streaming) — данные обрабатываются сразу после 
поступления, без ожидания накопления батча. Тяжёлые задачи, например обучение моделей и анализ сцен, 
выполняются отдельно в offline-контуре.

3. Каков допустимый уровень задержки?  
Допустимая задержка должна быть низкой, чтобы пользователь не замечал задержек при просмотре видео.
Система должна успевать обработать сегмент до окончания буфера воспроизведения. Если обработка занимает 
слишком много времени, используется fallback-механизм — пользователю показывается исходный видеосегмент 
без рекламной вставки. По этой причине пакетная обработка (batch) и микробатчинг здесь не подходят, так 
как они увеличивают задержку.

4. Каковы требования к отказоустойчивости?  
Система должна быть отказоустойчивой, потому что сбой не должен останавливать воспроизведение видео.
Для этого используется Kafka Event Log, где сохраняются все события системы. Если возникает ошибка, 
данные можно повторно обработать. Также можно добавить:
- резервирование online-сервисов;
- мониторинг состояния системы;
- fallback на оригинальный видеопоток при ошибке.

5. Какой уровень масштабируемости нужен?  
Система должна легко масштабироваться, так как количество пользователей и объём видео могут расти.
Масштабирование выполняется горизонтально:
- по видеосегментам;
- по GPU-воркерам для рендеринга;
- по online-сервисам обработки.
Это позволяет увеличивать производительность без полной перестройки системы.

6. Выбор архитектуры  
Для данной задачи выбрана Kappa-архитектура.
Причина выбора — работа с потоковыми данными и необходимость обработки в реальном времени. 
Все события проходят через единый журнал Kafka, а при изменении модели данные можно повторно обработать 
через replay.
Lambda-архитектура не выбрана, потому что она требует поддержки двух отдельных контуров обработки 
(batch и streaming), что усложняет систему и приводит к дублированию логики.
В данной задаче это избыточно.

7. Описание схемы архитектуры  
[Диаграмма](diagrams/vpp_simple.png)

| Блок                     | Назначение                                                                                       |
| ------------------------ | ------------------------------------------------------------------------------------------------ |
| **Sources**              | Источники данных: зритель, видеоконтент и рекламные кампании                                     |
| **Kafka Event Log**      | Центральный поток событий системы. Через него проходят запросы, сегменты видео и события показа  |
| **Ad Decision**          | Выбор рекламной кампании и подходящего места вставки с учетом таргетинга                         |
| **GPU Rendering**        | Встраивание бренда в видеосегмент                                                                |
| **CDN Packager**         | Подготовка и доставка персонализированного видеопотока зрителю                                   |
| **Spark (Offline Path)** | Обработка накопленных данных: анализ сцен, обучение моделей, повторная обработка данных (replay) |
| **Scene Inventory**      | Хранение информации о доступных местах вставки брендов в сценах                                  |
| **Monitoring**           | Контроль задержек, качества рендеринга и состояния системы                                       |

8. Пример последовательности обработки  
Работа системы происходит потоково, по мере поступления видеосегментов.
- Видеопоток разбивается на сегменты (например, HLS/DASH сегменты по несколько секунд), а информация о 
зрителе и рекламной кампании поступает в Kafka Event Log.
- Каждый сегмент сразу обрабатывается в streaming-режиме, без ожидания накопления батча.
- Ad Decision определяет, какую рекламную кампанию показать и в какое место сцены можно встроить бренд.
- Для проверки доступных мест вставки используется Scene Inventory, где заранее сохранена информация о 
  безопасных рекламных слотах.
- GPU Rendering встраивает выбранный бренд в текущий видеосегмент.
- CDN Packager подготавливает персонализированный сегмент и сразу отправляет его зрителю.
- Параллельно Spark в offline-контуре анализирует накопленные данные из Kafka: переобучает модели, 
  анализирует качество вставок и при необходимости запускает replay обработки.

9. Итог  
Для задачи динамической вставки брендов в видеопоток оптимально подходит Kappa-архитектура. 
Она хорошо работает с потоковыми данными, обеспечивает низкую задержку, позволяет повторно обрабатывать 
события и масштабируется при росте нагрузки. Тяжёлые вычисления выносятся в offline-контур, 
а online-часть отвечает за быструю обработку и показ персонализированного видео.
