# 2. metrics-recommendation

2026-05-15

## Status

Proposed

## Context
Если в целом говорить про online-кинотеатры ML-система используется там для персональных 
рекомендаций фильмов и сериалов. Для бизнеса важно, чтобы рекомендации увеличивали клики, просмотры, 
оплату, удержание пользователей и окупались. Для разработки и DevOps важно, чтобы сервис рекомендаций 
быстро отвечал, выдерживал пиковую нагрузку и не требовал лишней инфраструктуры без доказанной необходимости.  
Для ML-команды важно, чтобы модель хорошо ранжировала контент и не деградировала после выката в production.
Но самое важное в проекте есть конфликт требований и непонятных хотелок

| Участник | Типичный вопрос / требование |
|---|---|
| Бизнес | Нужно увеличить выручку, клики, просмотры и удержание |
| Разработчики | Нужен ли новый сервер, ClickHouse, переписывание backend |
| ML-команда | Какие метрики качества модели считать, нужен ли GPU, сколько рекомендаций отдавать |
| DevOps | Выдержим ли 10 000 RPS, какая latency допустима, не простаивают ли ресурсы |

Но решения по системе должны приниматься не по ощущениям и не от желания добавить новый сервис, а от реальной
необходимости, которая подтверждена конкретными метриками.

Примеры:

| Вопрос | На какие метрики смотрим |
|---|---|
| Нужен ли новый сервер? | Peak RPS, p95/p99 latency, CPU/RAM utilization, очереди, Error Rate |
| Нужен ли GPU? | Training time, inference latency, GPU utilization|
| Нужно ли переходить на ClickHouse? | DB load, latency SQL-запросов, объем событий, характер нагрузки|
| Нужно ли внедрять Kubernetes? | Availability, Deployment Stability, потребность в autoscaling |
| Хорошо ли работает модель? | Precision@K, Recall@K |
| Есть ли деградация модели? | Feature Drift Rate, Concept Drift, падение online-метрик |

## Decision

Использовать дерево метрик как основной механизм принятия решений по ML-системе.

```text
Цель:
увеличить бизнес-эффект онлайн-кинотеатра за счет рекомендаций
│
├── 1. Бизнес-метрики
│   ├── CTR
│   ├── Conversion Rate
│   ├── LTV
│   ├── ROI
│   └── Churn Rate
│
├── 2. Метрики разработки
│   ├── Availability
│   ├── Latency
│   ├── Throughput / RPS
│   ├── Error Rate
│
├── 3. ML-метрики

│   ├── Precision@K
│   ├── Recall@K

│
└── 4. DevOps-метрики
    ├── p95/p99 latency

    ├── CPU/GPU utilization
    ├── Availability
    ├── Error Rate    
```

## Metrics

### 1. Бизнес-метрики

| Метрика | Что показывает | Зачем нужна |
|---|---|---|
| **CTR** | Доля пользователей, кликнувших на рекомендацию. | Проверяет, интересны ли рекомендации пользователю. |
| **Conversion Rate** | Доля пользователей, совершивших целевое действие: просмотр, покупку, подписку или продление. | Проверяет, превращаются ли рекомендации в полезное бизнес-действие. |
| **LTV** | Доход от пользователя за все время использования сервиса. | Проверяет долгосрочную ценность пользователя. |
| **ROI** | Возврат инвестиций в ML-систему. | Проверяет, окупаются ли модель, инфраструктура и сопровождение. |
| **Churn Rate** | Доля пользователей, ушедших из сервиса. | Проверяет, не ухудшает ли система удержание. |


---

### 2. Метрики разработки

| Метрика | Что показывает | Зачем нужна |
|---|---|---|
| **Availability** | Процент времени, когда сервис рекомендаций работает корректно. | Требование к надежности API и fallback-логике. |
| **Latency** | Время обработки запроса, включая среднее и p95. | Требование к скорости backend-сервиса. |
| **Throughput / RPS** | Количество запросов в секунду. | Проверка пиковой нагрузки, например `10 000 RPS`. |
| **Error Rate** | Доля неудачных запросов, включая 4xx и 5xx. | Контроль качества backend, интеграций и обработки ошибок. |
| **Resource Usage** | Использование CPU, GPU, RAM, IO. | Проверка, нужен ли новый сервер или сначала нужна оптимизация. |

---

### 3. ML-метрики

Основная задача системы — рекомендации, поэтому базовые ML-метрики должны оценивать качество top-K выдачи.

| Метрика | Что показывает | Зачем нужна |
|---|---|---|
| **Hit Rate / Top-K Accuracy** | Попал ли релевантный фильм в top-K. | Простая проверка: нашла ли модель хотя бы один подходящий объект. |
| **Precision@K** | Долю релевантных объектов среди top-K. | Проверяет чистоту рекомендательной выдачи. |
| **Recall@K** | Долю найденных релевантных объектов среди всех релевантных. | Проверяет, не пропускает ли модель подходящий контент. |
| **PSI** | Изменение распределений признаков. | Контроль data drift. |
| **Feature Drift Rate** | Долю признаков с изменившейся статистикой. | Оценка масштаба изменения входных данных. |

Интерпретация PSI:

```text
PSI < 0.1     — стабильно
PSI 0.1–0.2   — умеренные изменения
PSI > 0.2     — серьезный drift
```

Для отдельных подмоделей классификации дополнительно используются:

```text
Accuracy
Precision
Recall
F1-score
ROC-AUC
LogLoss
```

---

### 4. DevOps-метрики

| Метрика | Что показывает | Зачем нужна |
|---|---|---|
| **p95/p99 latency** | Время ответа для 95% и 99% запросов. | Контроль задержек не только в среднем, но и на хвостах распределения. |
| **RPS** | Количество запросов в секунду. | Проверка готовности к пиковой нагрузке. |
| **CPU/GPU utilization** | Загрузку процессора и видеокарты. | Проверка, используются ли мощности эффективно. |
| **Availability** | Доступность сервиса. | Контроль выполнения SLO. |
| **Error Rate** | Долю ошибочных запросов. | Обнаружение проблем в сервисе, инфраструктуре или интеграциях. |

Базовые SLO:

```text
Latency p95 < 1 сек.
Error Rate < 1%
Availability > 99%
Peak Throughput: до 10 000 RPS
```

## Concrete SLO / KPI examples

Эти значения не являются финальными production-нормами, но задают понятные ориентиры для MVP и позволяют связать метрики с действиями команды.

| Группа | Метрика | Норма / целевой порог | Деградация | Действие |
|---|---|---:|---:|---|
| **Бизнес** | CTR по рекомендациям | `+5%` к baseline в A/B-тесте | нет роста или падение CTR | анализ выдачи, UX-блока и признаков модели |
| **Бизнес** | Conversion Rate | `+2%` к baseline | рост CTR без роста конверсии | пересмотреть целевую метрику и качество рекомендаций |
| **Бизнес** | Churn Rate | не выше baseline | рост `> 0.5 п.п.` | остановить раскатку, проверить влияние рекомендаций |
| **Бизнес** | ROI | `ROI > 1` после пилота | ROI `< 1` | не масштабировать инфраструктуру без оптимизации |
| **Разработка** | Latency p95 | `< 1 сек.` | `> 1 сек.` | профилирование backend, БД, кэша |
| **Разработка** | Error Rate | `< 1%` | `> 1%` | анализ ошибок, rollback или fallback |
| **Разработка** | Throughput / RPS | до `10 000 RPS` в пике | сервис не держит пик | нагрузочный тест, оптимизация, масштабирование |
| **Разработка** | Availability | `> 99%` | `< 99%` | incident review, fallback, исправление отказов |
| **ML** | Hit Rate@K | выше baseline-модели | падение относительно baseline | переобучение или откат модели |
| **ML** | Precision@K / Recall@K | не ниже baseline | стабильное падение качества | анализ датасета, признаков, разметки |
| **ML** | PSI | `< 0.1` | `> 0.2` | проверка data drift, переобучение модели |
| **ML** | Feature Drift Rate | `< 10%` изменившихся фичей | `> 20%` | анализ источников данных и пайплайна |
| **DevOps** | p95 latency | `< 1 сек.` | `> 1 сек.` | проверить saturation, очереди, сеть, БД |
| **DevOps** | p99 latency | `< 2 сек.` | `> 2 сек.` | анализ хвостовых задержек |
| **DevOps** | CPU/RAM utilization | `60–80%` под пиковой нагрузкой | `> 85%` или постоянный saturation | масштабирование или оптимизация |
| **DevOps** | GPU utilization | `> 50%`, если GPU используется | `< 30%` | пересмотреть необходимость GPU |
| **DevOps** | Cost per 1000 requests | не выше согласованного бюджета | рост стоимости без роста ROI | оптимизация инфраструктуры |

## Decision rules

### Бизнес-эффект рекомендаций

Продолжать развитие ML-рекомендаций и увеличивать инвестиции в систему имеет смысл только при подтвержденном бизнес-эффекте.

Смотрим на:

```text
CTR
Conversion Rate
LTV
ROI
Churn Rate
```

Решение:

```text
Если CTR и Conversion Rate растут, LTV увеличивается, Churn Rate не ухудшается,
а ROI положительный — ML-система дает бизнес-эффект.

Если растет только CTR, но нет роста Conversion Rate, LTV или ROI —
нужно пересматривать модель, UX выдачи или целевую метрику.
```

---

### Новый сервер

Новый сервер рассматривается только при подтвержденной нехватке ресурсов.

Смотрим на:

```text
Peak RPS
p95/p99 latency
CPU/RAM utilization
очереди
Error Rate
```

Решение:

```text
Если latency растет, RPS не достигает цели, CPU/RAM saturated и растут очереди —
масштабирование или новый сервер обоснованы.

Если CPU/RAM недогружены —
сначала проверяем backend-код, SQL-запросы, кэш, Docker-конфигурацию и сеть.
```

---

### GPU

GPU рассматривается только если он дает измеримый эффект.

Смотрим на:

```text
training time
inference latency
GPU utilization
NDCG@K / MAP@K / Hit Rate@K
ROI
```

Решение:

```text
Если GPU ускоряет обучение или inference, улучшает качество рекомендаций
и дает положительный ROI — использование GPU обосновано.

Если модель не использует GPU эффективно или бизнес-метрики не растут —
GPU не нужен.
```

---

### ClickHouse

ClickHouse рассматривается только при доказанной аналитической нагрузке.

Смотрим на:

```text
DB load
latency SQL-запросов
объем событий просмотров
характер запросов
RPS
```

Решение:

```text
Если Postgres не справляется с большими аналитическими запросами по событиям —
ClickHouse можно рассматривать.

Если проблема вызвана индексами, отсутствием кэша, плохими SQL-запросами
или Docker-in-Docker — сначала исправляем это.
```

---

### Kubernetes

Kubernetes рассматривается только при реальной эксплуатационной необходимости.

Смотрим на:

```text
Availability
Deployment Stability
частота релизов
число сервисов
потребность в autoscaling
```

Решение:

```text
Если сервисов много, нужны autoscaling, контролируемые релизы и выполнение SLO —
Kubernetes обоснован.

Если это простой MVP — Kubernetes может быть лишним усложнением.
```

## Consequences

### Положительные последствия

- Требования разных команд переводятся в измеримые показатели.
- Споры про инфраструктуру решаются через метрики, а не через мнения.
- Видно, какая метрика отвечает на какой вопрос.
- ML-система оценивается по бизнес-эффекту, качеству модели, надежности и стоимости.

### Риски

- Рост CTR может не привести к росту Conversion Rate, LTV или ROI.
- Offline-качество модели может не совпасть с production-эффектом.
- Data drift и concept drift могут снизить качество рекомендаций.
- Инфраструктура может подорожать быстрее, чем вырастет бизнес-эффект.
- Сервис может быть быстрым и стабильным, но бесполезным для бизнеса.
